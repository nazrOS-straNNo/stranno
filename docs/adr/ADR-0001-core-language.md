# ADR-0001: Язык ядра nazrOS CORE

**Статус**: Принято  
**Дата**: 2026-05-31  
**Авторы**: nazrOS Core Team

---

## Контекст

nazrOS CORE — фундамент всей платформы СТРАННО. Ядро управляет:
- Графом сцены (миллионы объектов)
- Шиной событий между модулями
- Менеджером GPU/CPU памяти
- Файловым форматом `.нзр`
- Системой undo/redo

Выбор языка определяет производительность, безопасность и сложность поддержки
на следующие 10+ лет.

---

## Рассмотренные варианты

### Вариант A: C++
- ✅ Максимальная производительность
- ✅ Зрелая экосистема (Unreal, Blender, Maya — все на C++)
- ✅ Прямой доступ к GPU API (Vulkan, Metal, DirectX)
- ❌ Undefined behavior, memory leaks, race conditions
- ❌ Сложный toolchain, медленная компиляция
- ❌ Сложный онбординг новых разработчиков

### Вариант B: Rust (основной)
- ✅ Безопасность памяти без GC (критично для real-time)
- ✅ Производительность сравнима с C++
- ✅ Отличный toolchain (Cargo, rustfmt, clippy)
- ✅ WASM-совместимость (веб-превью без переписывания)
- ✅ Растущая экосистема GPU (wgpu, ash)
- ⚠️ Steeper learning curve для C++ разработчиков
- ⚠️ Меньше готовых биндингов к индустриальным форматам (FBX и т.д.)

### Вариант C: Go
- ✅ Простота, скорость разработки
- ❌ GC — неприемлемо для real-time рендеринга и audio DSP
- ❌ Нет прямого GPU-доступа

---

## Решение

**Основной язык ядра: Rust**

**GPU-интенсивные модули (DSP, низкоуровневый Vulkan): C++ с FFI-биндингами из Rust**

Конкретно:
- `nazrOS CORE`, `Scene Graph`, `файловые форматы`, `шина событий` → **Rust**
- `ШИНА` (audio DSP на JUCE) → **C++** с Rust-обёрткой
- `Vulkan рендер-пайплайн` → **C++/GLSL** с вызовами из Rust через `ash` или `vulkano`
- UI shell → **Tauri** (Rust backend + React/TS frontend)

---

## Последствия

- Команда должна владеть Rust (минимум 2 senior Rust-инженера)
- FFI-граница Rust↔C++ требует документирования и тестирования
- Для биндингов к FBX/Alembic использовать C-библиотеки через `bindgen`
- Все публичные API модулей экспортируются через `extern "C"` для совместимости
